# 故障排查手册

## 1. 打开页面后直接停在启动错误页

优先检查：

- `public/app-config.json` 是否存在
- `apiBaseUrl` 是否是相对路径或合法 `http(s)` 地址
- `enabledModules` 字段是否完整且类型正确

这类错误会在业务页挂载前直接阻断启动。

## 2. 登录失败

优先检查：

- `site-gateway/site-config.json` 中的 `bootstrapUsers`
- 本地 SQLite 是否已经初始化过旧账号
- `.tmp/frontend-prod/frontend.err.log` 是否有站点网关报错

如果是首次部署，请确认站点网关用户表为空，或者按现场流程重置本地数据库。

## 3. 生产脚本启动失败

优先检查：

- 是否在发布目录执行过 `npm.cmd install --omit=dev`
- `dist/index.html` 是否存在
- 端口 `4173` 是否被其他程序占用
- `.tmp/frontend-prod/frontend.out.log`
- `.tmp/frontend-prod/frontend.err.log`

当前生产启动脚本不再自动执行 `verify`。如果是源码仓库直接启动，请先手工执行：

```powershell
npm.cmd run build
```

## 4. Windows 服务安装失败

优先检查：

- 是否提供了有效的 `-WinSwExePath`
- `node` 是否已安装并在 `PATH` 中
- 当前 PowerShell 是否有管理员权限
- `service/clean-robot-site-service.xml` 是否已生成

如果服务名称已存在，可先执行：

```powershell
.\scripts\uninstall-site-service.ps1
```

再重新安装。

## 5. 页面能打开，但始终显示 ROS 未连接

优先检查：

- `site-gateway/site-config.json` 中的 `rosbridgeUrl`
- 现场 rosbridge 服务是否启动
- 目标地址是否可从现场主机访问
- 防火墙是否拦截 websocket 端口

可以先访问：

```text
http://127.0.0.1:4173/api/health
```

看返回中的 `ros.status` 和 `ros.lastError`。

## 6. Mock 正常，Live 异常

这通常说明：

- 前端页面壳层本身没有问题
- 问题更可能在现场 ROS、站点网关配置或依赖服务

建议顺序：

1. 先跑 `npm.cmd run test:e2e`
2. 再看 `/api/health`
3. 再看总览页能力状态
4. 最后导出诊断包交给研发分析

## 7. 高风险按钮被阻断

这通常是正常的权限或依赖校验结果。

检查：

- 当前登录角色是否具备对应 capability
- 站点网关是否返回 `requiresEngineer=true`
- 目标 ROS service 是否存在
- 总览页最近审计记录里是否有 `blocked` 事件

## 8. 升级脚本失败

优先检查：

- `InstallRoot` 和 `PackageRoot` 是否是不同目录
- `BackupRoot` 是否错误地放在安装目录里面
- 当前服务是否已先停掉
- 新发布包是否包含完整 `dist/`、`site-gateway/` 和 `scripts/`

如果升级中断，先查看备份目录，再决定是否执行回滚：

```powershell
.\scripts\rollback-site-release.ps1 -InstallRoot C:\CleanRobot\site
```

## 9. 如何安全停机

优先使用：

```powershell
.\stop-frontend-prod.cmd
```

如果走服务化部署，也可以先停服务，再做升级或排障。不要直接手工杀掉未知 PID。

如果脚本提示 `Access is denied`，说明当前终端没有权限停止占用站点端口的进程。请改用管理员 PowerShell 重新执行停机脚本，或先定位占用 4173 的 PID：

```powershell
Get-NetTCPConnection -State Listen | Where-Object { $_.LocalPort -eq 4173 }
```

确认该 PID 确实是清洁机器人站点网关后，再由管理员终端停止。旧 `release/` 目录里的 `site-gateway.sqlite` 被占用时，也先按这个流程停掉站点网关，再执行：

```powershell
npm.cmd run clean:workspace:release
```

## 10. Ubuntu 小车开机停在“需要认证”

现象：

- 开机后进入 Ubuntu 桌面，但没有直接进入前端
- 弹出“需要认证 / 您登录计算机时，您的登录密钥环未被解锁”
- 输入系统密码后 kiosk/Chromium 才继续打开

这通常不是前端错误，而是自动登录后 GNOME login keyring 没有被解锁。

处理顺序：

1. 确认小车触摸屏用户已经启用自动登录，常见配置位置是 `/etc/gdm3/custom.conf`
2. 确认 kiosk Chromium 启动参数包含 `--password-store=basic`
3. 确认 kiosk Chromium 使用独立 profile，例如 `--user-data-dir="$HOME/.config/clean-robot-kiosk-chromium"`
4. 如仍弹窗，备份并重建当前触摸屏用户的 login keyring，或通过 Passwords and Keys 将 Login keyring 改为空密码

命令行处理前先备份：

```bash
mkdir -p ~/.local/share/keyrings.backup
cp -a ~/.local/share/keyrings/* ~/.local/share/keyrings.backup/ 2>/dev/null || true
```

然后清理当前用户旧 keyring：

```bash
rm -f ~/.local/share/keyrings/login.keyring
rm -f ~/.local/share/keyrings/user.keystore
```

重启后确认不再弹出认证框。不要为了规避该弹窗而给触摸屏用户配置全局 sudo 免密。

## 11. Ubuntu kiosk 没有自动打开前端

优先检查：

```bash
systemctl status clean-robot-site-gateway --no-pager
journalctl -u clean-robot-site-gateway -n 80 --no-pager
curl http://127.0.0.1:4173/api/health
ls -l ~/.config/autostart/
ls -l ~/.local/bin/clean-robot-kiosk.sh
```

如果 Site Gateway 正常但没有打开浏览器，检查：

- `~/.config/autostart/clean-robot-kiosk.desktop` 是否存在
- `Exec=` 路径是否是当前触摸屏用户的真实路径
- `clean-robot-kiosk.sh` 是否有执行权限
- Chromium 命令是 `chromium-browser`、`chromium` 还是 `google-chrome`
- kiosk 脚本是否等待 `/api/health` 成功后再启动浏览器

进入桌面维护或临时退出 kiosk：

```bash
pkill -f clean-robot-kiosk.sh || true
pkill -f "chromium.*--kiosk" || true
pkill -f "chrome.*--kiosk" || true
```

临时禁止下次开机自启动：

```bash
mv ~/.config/autostart/clean-robot-kiosk.desktop ~/.config/autostart/clean-robot-kiosk.desktop.disabled
```

恢复时改回：

```bash
mv ~/.config/autostart/clean-robot-kiosk.desktop.disabled ~/.config/autostart/clean-robot-kiosk.desktop
```

## 12. `4173` 健康检查正常但页面白屏

如果 `http://127.0.0.1:4173/api/health` 正常，但页面白屏，优先看浏览器 Console。

常见原因：

- 发布包中的生产构建 chunk 运行时异常
- kiosk 浏览器缓存了旧的 `dist/assets/*.js`
- 部署时混用了旧 release 的 `dist/` 和新 release 的 `site-gateway/`

处理原则：

- 不要在小车端手动热修 `dist/assets/*.js` 作为最终交付方案
- 回到 Windows 前端源码工程修复构建配置，重新执行 `npm.cmd run package:trial`
- 小车端用完整新 release 替换旧 release，再重启 `clean-robot-site-gateway`
- 必要时清理 kiosk 独立 profile 缓存：`~/.config/clean-robot-kiosk-chromium`

验收时至少确认：

- 登录页 DOM 正常挂载
- DevTools Console 不再出现 vendor chunk 初始化异常
- 刷新、强制刷新、重启 kiosk 后都不白屏

## 13. 小车重启后跳过登录页

现象：

- 关机前触摸屏已经登录前端
- 小车断电重启后直接恢复到总览页或其他业务页
- 关机前如果处于登录页，重启后又能正常显示登录页

原因通常是旧登录会话被保留：

- Chromium kiosk profile 保留了登录 cookie
- Site Gateway 的 `.tmp/site-gateway/site-gateway.sqlite` 中还存在未过期 session

当前商用默认行为应为：

```json
"clearSessionsOnStartup": true
```

检查小车部署目录：

```bash
cd /opt/clean-robot-site/current
grep -n "clearSessionsOnStartup" site-gateway/site-config.json
```

如果缺少该字段或为 `false`，改为：

```json
"clearSessionsOnStartup": true
```

然后重启服务：

```bash
sudo systemctl restart clean-robot-site-gateway
```

再刷新 kiosk 页面或重启小车。期望结果是回到前端登录页。只有内部调试明确需要保留重启前登录态时，才允许把该字段改为 `false`。
